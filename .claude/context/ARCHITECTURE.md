## Data Boundaries — Three Independent Prisma Schemas

The monorepo has **three separate `schema.prisma` files, each its own bounded context with its own generated client**. They are NOT one shared database split by tooling — they are independent data stores.

| Schema file | Owner | Generated client | Models | Datasource |
| ----------- | ----- | ---------------- | ------ | ---------- |
| `packages/database/prisma/schema.prisma` | `@core/database` (consumed by `internal/api-example` in `DATA_SOURCE=prisma` mode) | `packages/database/src/generated/prisma` | `Party`, `User`, `Role`, `UserRole` (reference model) | `postgresql` |
| `apps/api-iam/prisma/schema.prisma` | `@corehub/api-iam` (:8080) | `apps/api-iam/src/generated/prisma` | ~30 models: `Tenant`, `User`, `Plan`, `OnboardingDraft`, `Payment`, `Invitation`, … + tenant schemas (`tenant_{slug}` via `SET search_path`) | `postgresql` (DB on :5432) |
| `apps/api-instagram-analytics/prisma/schema.prisma` | `@corehub/api-instagram-analytics` (:3003) | default (`node_modules/.prisma`, `prisma-client-js`) | `InstagramAccount`, `InstagramMedia`, `Carousel`, `ContentSuggestion`, `AiUsageLog`, … | `env("DATABASE_URL")` |

### The boundary rule (critical)

- **No cross-schema foreign keys.** `api-instagram-analytics` stores `tenantId` and `userId` as **opaque strings taken from the verified JWT** (`tenant_uuid` / `sub` claims) — never as a Prisma relation to `api-iam`'s `Tenant`/`User`. The identity boundary is the JWT, not the database. This is correct microservice isolation: each service owns its data, and the only contract between them is the token.
- **`api-iam.User` and `@core/database.User` are different models in different clients.** They share a name but are unrelated. Never import one expecting the other; always import from the owning package's generated client.
- **Physical DB relationship is deployment config, not code.** `api-iam` and `api-instagram-analytics` MAY share one PostgreSQL instance (their table names don't collide — `InstagramAccount` vs `tenants` etc.) or run on separate instances. Recommended: separate databases (or at least separate schemas) so the bounded-context boundary is physical, not merely nominal. Whichever is chosen, document it in each app's `.env.example` `DATABASE_URL`.

### Migration drift caveat

Because the schemas are independent, **migrations are per-app** (`pnpm --filter @corehub/api-iam db:migrate`, etc.) and never coordinated automatically. `api-iam` already logged one drift incident (`20260513200000_fix_schema_drift`) where the Prisma schema diverged from the SQL migrations. Each service is responsible for keeping its own schema↔migration↔generated-client in sync; there is no monorepo-wide migration gate.

## @core/cli — Architecture

**Entry**: `packages/cli/src/index.ts` — commander program
**Commands**:

- `core new api [name]` — scaffolds Hono API into `apps/{name}/`
- `core new webapp [name]` — scaffolds Next.js webapp into `apps/{name}/`
- `core add route|page|component` — adds to existing app
- `core docs` — documentation management

**Generation flow**:

1. Prompt for missing options (port, database, auth, theme)
2. Read `templates/{api|webapp}/manifest.json` for file list
3. Process each `*.template` file via variable substitution
4. Write to target `apps/{name}/` directory

**Template variables**: `{{NAME}}`, `{{PORT}}`, `{{DATABASE}}`, `{{AUTH}}`, `{{API_URL}}`, etc.

**Port allocation**: APIs start at 3010, webapps start at 3021 (auto-increments by scanning existing apps' `package.json`)

## apps/api-iam — Identity & Access Management

**Status**: Implementado (SDDs `iam-bootstrap` + `iam-logging` archivados)
**Port**: 8080 | DB (Docker): 5432 | Adminer: 8081

**Stack**: Hono 4.x + `@hono/zod-openapi` + Prisma 6.x + PostgreSQL 16 + JWT RS256 (`jose`) + argon2id + Pino v10

**31 endpoints**: Auth (14), Onboarding (9), Webhooks (1), Identity (1), Plans (2), Invitations (2), Billing (1), Well-known (1), Health (1)

```
src/
├── index.ts                   # Composition root — async function main() + await createLogger(config)
├── config.ts                  # Env vars via Zod (⚠️ line 93 uses console.* — logger not ready yet)
├── errors.ts                  # AppError hierarchy + Problem Details (RFC 7807)
├── domain/                    # Pure TypeScript interfaces
├── repositories/              # Prisma implementations (12 repositories)
├── services/                  # Business logic (14 services, all receive logger: Logger in deps)
├── routes/                    # Hono routes by resource (auth, onboarding, plans, invitations, identity, billing, webhooks, well-known, health)
├── middleware/                # error-handler, request-id, auth-guard, idempotency, tenant-guard
│   └── access-log.middleware.ts  # HTTP access log (Pino, category: 'http')
├── adapters/                  # Swappable: OTP, Email, Bancard, PDF, Storage, RateLimiter, KeyProvider
├── lib/
│   ├── logger.ts              # createLogger(options): Promise<Logger> — leaf module (no service imports)
│   └── index.ts               # Barrel
├── jobs/
│   └── background-jobs.ts     # 5 cron jobs: cleanup OTP, password reset, idempotency, refresh tokens, drafts
├── test-helpers/
│   └── logger.ts              # silentLogger + createMemoryLogger() for tests
├── db/
│   ├── migration-runner.ts    # SQL migration runner for tenant schemas
│   ├── with-tenant.ts         # SET search_path helper for multi-tenant queries
│   └── migrations-tenant/     # SQL files per tenant schema version
└── scripts/
    └── generate-dev-keys.ts   # RSA key generation for development
```

**Key architectural decisions**:
- Schema-per-tenant: each tenant has a PostgreSQL schema `tenant_{slug}`. Queries use `SET search_path TO tenant_{slug}` via `withTenant()` helper, never hardcoded.
- DI: manual top-down, no IoC container. Composition root wires all deps and passes them down.
- Logging: Pino v10, `logger: Logger` is required (not optional) in all `*Deps` interfaces. Category field differentiates event domains: `http`, `auth`, `otp`, `job`, `system`. `refresh_reuse` logged at ERROR level (token theft signal).
- Mock strategy: `vi.clearAllMocks()` in tests (NOT `vi.restoreAllMocks()` — breaks module mocks like argon2).
- Docs: `apps/api-iam/README.md`, `.atl/runbook.md`, `.atl/jwt-key-rotation.md`, `.atl/manual-tenant.md`, `.atl/logging-guide.md`

**Onboarding — Draft Recovery** (`PATCH /onboarding/draft/:draftId/recover`):
- Public endpoint (no auth guard), no idempotency middleware
- Allows resetting `data.company` from a stuck `payment_confirmed` draft (when RUC/email conflict)
- Guard: draft must have `status=payment_confirmed` AND `tenantId=null` (not yet provisioned)
- `data.company` is cleared → `deriveCurrentStep` on hub redirects wizard back to company step automatically

**Schema — Known Drift Fix** (migration `20260513200000_fix_schema_drift`):
- `activation_token_hash` (VARCHAR(64), unique), `activation_token_expires_at` (TIMESTAMP(3)), `activation_token_used` (BOOLEAN DEFAULT false) — were in Prisma schema but missing from migrations
- `refresh_tokens.family_id` changed from UUID to VARCHAR(32) to accept nanoid() values (21 chars)
- Submit service: `tokenFamilyId = nanoid()` — NOT `crypto.randomUUID()` — familyId is @db.VarChar(32)

**Prisma Schema — Key Models**:

| Model | Purpose |
|-------|---------|
| `InstagramAccount` | OAuth account per tenant+user. Stores `tokenEncrypted` (AES-256-GCM), `igUserId`, `syncStatus`. Unique on `(tenantId, userId)`. |
| `InstagramMedia` | Synced media items (posts, reels, stories). |
| `InstagramMediaMetric` | Metrics per media per sync: likes, comments, saves, shares, reach, videoViews, etc. |
| `InstagramAccountInsight` | **One record per sync execution** (NOT per calendar day). |
| `InstagramSyncLog` | Audit log per sync. |
| `ChatMessage` | Growth agent conversation history per session. |
| `ContentSuggestion` | AI-generated content ideas. `batchId` agrupa sugerencias de una misma interacción. `category`: caption/format/posting_time/hook/hashtags/content_idea. Solo `content_idea` puede generar carrusel. |
| `SuggestionBatch` | Agrupa sugerencias generadas en una misma interacción o request. `userMessage` = prompt original. |
| `InstagramAgentConfig` | Per-account agent settings: niche, tags, customPrompt, imageGen config. |
| `Carousel` | Pipeline state: status (pending→generating_script→generating_images→ready/failed), publishStatus (unpublished/published/failed). |
| `CarouselSlide` | Slide individual: order, role (hook/development/cta/default), text (español), visualPrompt (inglés), imageUrl (ruta relativa /carousels/...). |

**Key Architectural Decisions**:

1. **Instagram API v25.0**: Account-level metrics REQUIRE `metric_type=total_value`. Exception: `follower_count` WITHOUT `metric_type`.

2. **Carousel image URL management** (dos capas):
   - Backend `DiskImageStorage` siempre guarda **rutas relativas** (`/carousels/{id}/{slideId}.jpg`)
   - `resolvePublicUrl()` en `CarouselService` SIEMPRE extrae el pathname y rebasa a `PUBLIC_BASE_URL` (ngrok) — nunca early-return para URLs absolutas (bug anterior: pasaba localhost a Instagram)
   - Frontend `resolveImageUrl()` rebasa cualquier URL a la URL local del API para display (`http://localhost:3003`)

3. **Instagram Publishing — container readiness**: Después de `createCarouselContainer()` hay que llamar `waitForContainerReady()` (polling cada 3s, timeout 90s) antes de `publishMedia()`. Publicar inmediatamente causa error 9007 "Media ID is not available".

4. **Carousel generation — previewScript flow**: `POST /preview-script` es un endpoint stateless que llama a DeepSeek sin persistir nada. Solo cuando el usuario aprueba, `createCarousel` se llama con `approvedSlides` — saltándose la generación de script y yendo directo a imágenes. Evita crear registros en DB para previews.

5. **Script generation — idiomas**: DeepSeek genera `text` en español y `visualPrompt` en inglés. fal.ai/ideogram requiere prompts en inglés. Los prompts visuales deben describir el contenido del slide (no solo estética genérica).

6. **SuggestionService.generateContentIdea()**: Requiere `DeepSeekClient` como 2° argumento del constructor. Crea un `SuggestionBatch` con el prompt como `userMessage` (para que la sugerencia aparezca agrupada en la vista de batches del frontend). Tests existentes no lo pasan (es opcional).

7. **`InstagramAccountInsight` — one record per sync, not per day**.

8. **`media_url` — URL temporal de CDN**: Nunca se almacena. Fetcheada on-demand vía `/api/media/:id/playback`.

9. **Rate limit**: ~200 llamadas/hora por token. Auto-retry con backoff exponencial en `withRetry()`.

**API Response Envelope**: igual al patrón del monorepo — `{ success: true, data: ... }` / `{ success: false, error: { code, message } }`.

## apps/hub — Token Lifecycle & Known Gotchas

### Access token storage
- Module-level singleton: `apps/hub/src/modules/identity/session/token.ts` → `currentToken: AccessToken | null`
- `setAccessToken(token)` / `getAccessToken()` — reference equality, new object on every set
- `apiFetch` reads `getAccessToken()` before every request; skips Authorization header if null or expired (30s skew)

### Race condition: AuthProvider refresh vs submitDraft (fixed)
**Scenario**: After the Bancard stub redirect (`localhost:8080 → localhost:3001`), the hub does a full page reload. `AuthProvider` mounts and immediately calls `refreshSession()` with no token. Meanwhile, payment polling navigates to /summary and `submitDraft` sets a new access token. When `refreshSession()` fails (no refresh cookie yet), it used to call `clearAccessToken()` — wiping the token `submitDraft` just set. This caused 401 on the billing/documents endpoint when the user clicked download.

**Fix** (`refresh.ts:executeRefresh`): snapshot `tokenAtStart = getAccessToken()` before the fetch. On failure, only call `clearAccessToken()` if `getAccessToken() === tokenAtStart`. Reference equality ensures a token set by `submitDraft` while refresh was in-flight is preserved.

### Bancard stub — full page navigation
The Bancard stub returns `redirectUrl: 'http://localhost:8080/__stub/bancard/approve?...'`. Because this is a different origin from the hub (`localhost:3001`), `PaymentInitiateView` uses `window.location.href` (not `router.push`). This causes a **full page reload** → all module-level state (access token, session) is cleared. The flow reconstitutes itself via:
1. `AuthProvider` silent refresh (fails if no prior session cookie — expected during onboarding)
2. Payment polling navigates to /summary
3. `submitDraft` re-establishes the token after provisioning
